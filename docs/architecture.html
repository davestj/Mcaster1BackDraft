Mcaster1BackDraft C++ Daemon (port 9432)
    │
    ├─ 1. Parse HTTP request (method, path, query, headers, body)
    ├─ 2. Extract X-Real-IP, X-BackDraft-Page
    ├─ 3. URL-decode query string and path
    ├─ 4. Load WAF rules from MySQL (cached, reloaded every 30s)
    ├─ 5. Evaluate each rule: regex/contains/equals matching
    ├─ 6. Calculate threat score (sum of matched rule scores)
    ├─ 7. Determine action: pass / log / flag / block
    ├─ 8. Log request to backdraft_requests table
    ├─ 9. If score >= threshold: log to backdraft_threats
    ├─ 10. If mode=active && action=block: return 403
    │
    ├─ 11. Look up doc_root from backdraft_sites by Host header
    ├─ 12. Build FastCGI request (FCGI_BEGIN + FCGI_PARAMS + FCGI_STDIN)
    ├─ 13. Connect to PHP-FPM unix socket
    ├─ 14. Send FastCGI request, read FCGI_STDOUT response
    ├─ 15. Parse PHP response headers (Status, Content-Type)
    │
    ▼
Return HTTP response to nginx → Client

Thread Pool

Each of the 3 HTTP listeners gets its own bounded thread pool. Pool size is auto-calculated:

  • Base: CPU cores * 4 (I/O-bound work: FastCGI waits on PHP-FPM)
  • Capped by memory: available_MB / 2 / 2 (each thread ~2MB stack)
  • Floor: 8 threads minimum
  • Ceiling: 512 threads maximum

On a 4-core server with 8GB RAM: 16 threads per port = 48 total worker threads.

Database Schema

16 tables in mcaster1_backdraft. Key tables:

  • backdraft_requests — HIGH VOLUME. Every WAF-inspected request: IP, method, path, query, user agent, threat score, action, matched rules, upstream status, latency.
  • backdraft_threats — Elevated events (score >= threshold): category, severity, matched rules, dismissal status.
  • backdraft_rules — WAF rules: target, operator, value (regex/contains/equals), action, score. Supports builtin, custom, learned, imported types.
  • backdraft_sites — Discovered nginx sites: server_name, config_path, doc_root, access/error log paths, WAF/log enrollment status.
  • backdraft_perf_metrics — Self-monitoring snapshots every 10s: CPU, RSS, threads, FDs.

Why Not Go?

The initial design used Go sidecars for the WAF proxy and log daemon. This was scrapped because:

  • Go's httputil.ReverseProxy speaks HTTP, not FastCGI — creating an impossible proxy loop through nginx
  • The C++ daemon already has a working FastCGI client
  • Multiple binaries across languages = multiple build systems, multiple deploy targets, multiple failure modes
  • One C++ binary with 3 listeners is simpler to develop, test, deploy, and manage
